# LiveChat 智能分配与路由重构 MVP

## 1. 背景

当前系统已经具备以下基础能力：

- 访客聊天页、客服工作台、消息收发、转接、结束会话
- 客服在线状态感知
- `agent registry` 的能力标签、容量上限、分配与释放
- Kafka 异步调度链路，可用于削峰和异步消费

当前主要限制：

- 访客入口依赖 `/livechat?user_id=<客服账号>`，必须显式指定客服
- 人工客服分配仍以“固定绑定客服”为主，不是统一队列分配
- 自动分配只在“人工客服离线且自动回复未命中”时才触发，属于兜底，不是主流程
- 读写模型仍以单链路直写为主，尚未区分消息主存、工作台视图和异步消费视图

本次不是只做局部功能补丁，而是以主流客服系统为参考，对接待入口、会话路由、客服状态、消息事件和工作台视图做一轮架构级重构。

本 MVP 的目标，是把系统从“指定客服接待”升级到“统一入口 + 智能分配 + 队列路由 + 可扩散的事件链路”。

## 2. MVP 目标

MVP 版本聚焦五件事：

1. 访客进入聊天页时不再强制输入客服账号。
2. 系统能根据规则自动分配一个在线且可接待的人工客服。
3. 分配失败时支持排队、超时扩散和兜底提示。
4. 消息链路明确区分“主消息存储”和“工作台/统计/自动化消费”的扩散方式。
5. 在不丢失现有能力的前提下，完成一次围绕客服路由中心的整体重构。

## 3. 对标原则

本次设计明确要求参考主流客服系统思路，不以“能跑通”为唯一标准，而以“是否接近成熟客服系统的组织方式”为标准。

对标原则如下：

- 队列优先，不再以“访客直连某个客服账号”为系统主模型
- 可接待不等于在线，在线状态、忙碌状态、容量状态要分离
- 分配器是独立中枢，不把路由逻辑散落在控制器、WebSocket 和消息发送逻辑中
- 默认采用 `sticky_first + least_loaded`，而不是简单轮询
- 队列、owner、历史接待客服、机器人接待方必须区分语义
- 消息正文单写，读模型和工作台视图扩散
- 所有自动分配都必须支持人工干预、转接、释放、回收和超时扩散
- 路由与消息事件都要可观测、可重试、可回放
- 架构上支持后续接入 SLA、优先级、机器人协同、质检、统计和总结能力

## 4. 架构重构目标

本次重构不只是增加一个“自动选客服”函数，而是要形成清晰的客服系统架构。

### 4.1 重构目标

- 把“接待入口”和“客服 owner”彻底解耦
- 把“人工客服路由”从控制器和 WebSocket 分支逻辑中抽离出来
- 把“客服状态管理”从即时连接状态升级为可路由状态
- 把“消息主存”和“工作台视图”拆成两个层次
- 把 Kafka 从可选异步调用，升级为未来事件扩散的统一通道之一
- 让工作台展示的数据来自明确的会话读模型，而不是各处临时拼接

### 4.2 目标架构分层

建议形成以下层次：

- 接入层：访客登录、聊天页入口、工作台入口、兼容旧参数
- 路由层：队列、技能、容量、粘性、扩散、转接、释放
- 会话层：会话 owner、排队状态、生命周期、分配结果
- 消息层：消息主存、消息事件、推送分发
- 读模型层：工作台摘要、未读、最近消息、客服负载、排队视图
- 支撑层：Kafka、监控、日志、重试、超时、降级

### 4.3 模块边界

后续代码实现时建议遵守以下边界：

- `controller` 负责入参校验和返回，不直接决定分配策略
- `ws` 负责连接和推送，不直接持有复杂路由决策
- `router/dispatcher/service` 负责客服分配、释放、转接和扩散
- `models` 负责持久化，不承载复杂业务编排
- `Kafka` 负责事件投递与消费，不承担唯一事实来源

## 5. MVP 范围

### 5.1 In Scope

- 新增统一接待入口，不再要求访客显式传 `user_id`
- 新增人工客服路由层
- 引入客服路由中心，承载分配、转接、释放、扩散
- 支持在线状态、接待上限、技能标签、粘性分配、最小负载分配
- 支持分配超时后的扩散策略
- 支持客服手工转接
- 支持会话结束后释放客服容量
- 引入会话 owner 概念，`visitor.to_id` 从“入口指定客服”升级为“当前接待客服”
- 引入队列概念，支持 `pending` / `assigned` / `closed` 生命周期
- 明确消息主表、工作台摘要、事件总线的职责边界
- 沿用现有 Kafka，承担异步分配事件和消息事件扩散
- 工作台围绕“会话”和“队列”重组展示模型
- 进行必要的代码结构重构，而不是仅在原有函数上继续堆逻辑

### 5.2 Out of Scope

- 完整工单系统
- 多级审批、排班系统
- 基于机器学习的智能预测分单
- 跨团队复杂 SLA 编排
- 多租户后台隔离重构
- 一次性把所有消息存储迁移到全新事件溯源架构

## 6. 业务目标

- 访客发起会话时，无需知道具体客服账号
- 在线客服接待更均衡，减少人工抢单和手工转接成本
- 同一个访客再次进入时，优先回到最近接待客服
- 高峰期支持排队与扩散，不因单一客服不可用而阻塞
- 后续可以平滑接入机器人、质检、统计、总结等异步消费者
- 系统组织方式更接近主流客服系统，而不是继续围绕单一客服账号拼补丁

## 7. 核心方案

### 7.1 统一入口

访客入口从：

- `/livechat?user_id=<客服账号>`

演进为：

- `/livechat`
- `/livechat?entry_id=<入口标识>`
- `/livechat?service_line=<业务线>`

MVP 阶段建议兼容旧参数，但主模型切换为统一入口：

- 如果传了 `user_id`，按旧逻辑兼容
- 如果未传 `user_id`，走新的智能分配逻辑

### 7.2 分配对象

MVP 先聚焦人工客服，不改变现有 agent/机器人链路定位，但会统一定义角色边界。

- 人工客服是主接待方
- 机器人/agent 保留为欢迎语、自动回复、兜底或未来扩展能力
- 不建议直接把现有 `agent registry` 命名复用为人工客服中心，但其“能力 + 容量 + 分配 + 释放”的结构可以借鉴
- 访客会话必须区分“当前 owner 是人工客服”还是“当前由机器人兜底”

### 7.3 分配策略

MVP 采用以下顺序：

1. 粘性分配
2. 技能过滤
3. 在线过滤
4. 容量过滤
5. 最小负载分配
6. 超时扩散
7. 无人可接待时进入排队或返回提示

规则说明：

- 粘性分配：访客历史有最近接待客服，且该客服在线且未满载，则优先分配回原客服
- 技能过滤：按 `service_line`、语言、售前/售后等标签筛客服池
- 在线过滤：只选在线客服
- 容量过滤：仅选择 `active_sessions < max_sessions` 的客服
- 最小负载分配：优先选择当前会话数最少的客服
- 打平规则：负载一致时，优先最近未分配的客服，避免热点集中
- 超时扩散：例如先在技能组等待 10 秒，超时后扩散到默认公共池

### 7.4 MVP 支持的分配模式

- `sticky_first`：优先老客服
- `least_loaded`：默认模式，适合客服支持场景
- `round_robin`：保留为可选模式，适合销售分单或简单轮询

MVP 默认推荐：

- 人工客服主模式使用 `sticky_first + least_loaded`
- 非特殊业务场景不把 `round_robin` 作为默认方案

## 8. 主流客服系统边界要求

本节用于约束实现时最容易做偏的地方。

### 8.1 在线不等于可接待

必须区分至少以下状态：

- `online`
- `away`
- `busy`
- `offline`

路由判断不能只看 WebSocket 是否在线，还要看：

- 是否启用接单
- 当前负载是否已满
- 是否处于手工暂停接待状态

### 8.2 队列不等于 owner

必须区分：

- 会话当前所在队列
- 当前会话 owner
- 最近一次服务该访客的客服
- 当前是否由机器人兜底

不能继续把所有语义都压在 `visitor.to_id` 上，只能在 MVP 兼容期临时复用该字段。

### 8.3 粘性分配必须可失效

以下场景必须允许跳过老客服：

- 老客服离线
- 老客服满载
- 老客服被禁用
- 等待超过粘性分配阈值

### 8.4 自动分配必须可人工覆盖

无论自动分配多完整，系统都必须保留：

- 手工转接
- 强制释放
- 手工结束会话
- 临时停接
- 指定客服接管

### 8.5 优先级必须先于公平性

分配顺序不能只讲公平，还要支持：

- VIP 会话优先
- 投诉或紧急会话优先
- 等待时间过长的会话优先
- 即将超时的会话优先

MVP 可以先预留字段和排序口子，即使第一版不做完整 SLA。

### 8.6 幂等与重复消费必须被考虑

Kafka 或重试场景下，以下操作都必须设计为幂等：

- 分配
- 释放
- 转接
- 会话状态更新
- 工作台摘要更新

### 8.7 失败不能放大到主链路

以下失败不能直接阻塞访客主聊天链路：

- Kafka 暂时不可用
- 摘要视图更新失败
- 统计消费者失败
- AI 总结失败

主链路优先保证：

- 会话可建立
- 消息可落主存
- owner 状态一致

## 9. 排队与扩散

### 9.1 排队

当技能池内无可用客服时：

- 将访客会话标记为 `pending`
- 返回“正在为你分配客服，请稍候”
- 记录入队时间和队列标签

### 9.2 扩散

当等待达到阈值时：

- 第一阶段：扩散到同业务线的全部在线客服
- 第二阶段：扩散到默认公共客服池
- 第三阶段：仍失败则显示离线留言或机器人兜底

MVP 推荐默认阈值：

- `10s` 技能池内等待
- `30s` 扩散到公共池

### 9.3 转接

客服手工转接保留，且转接后：

- 更新会话 owner
- 更新访客 `to_id`
- 向新客服推送在线会话
- 向访客发送“已转接”提示

## 10. 读扩散 / 写扩散设计

MVP 不追求完全事件溯源，但要先把边界定清楚。

### 10.1 单写主存

聊天消息正文只保留一份主记录。

目标：

- 避免一条消息按客服、报表、未读视图重复落多份
- 历史消息查询仍以消息主表为准

适用对象：

- `message` 主表
- 会话原始事件流

### 10.2 写扩散

对高频读取、低计算容忍的数据，使用写扩散维护视图。

适用对象：

- 客服工作台会话列表摘要
- 最近一条消息
- 未读数
- 当前 owner
- 排队状态

目标：

- 降低工作台首页查询联表成本
- 降低在线会话列表刷新压力

### 10.3 读扩散

对可异步处理的能力，基于主消息或事件流做读扩散。

适用对象：

- 统计报表
- AI 总结
- 关键词质检
- 告警通知
- 审计与归档

目标：

- 不阻塞主会话链路
- 允许消费者独立扩展

### 10.4 MVP 原则

- 消息正文：单写
- 工作台视图：写扩散
- 统计/机器人/质检：读扩散

## 11. 数据模型建议

MVP 不要求一次性大改表，但至少要补齐以下概念。

### 11.1 客服运行时状态

建议新增或抽象出客服运行时视图，包含：

- `kefu_id`
- `display_name`
- `online_status`
- `enabled`
- `skills`
- `max_sessions`
- `active_sessions`
- `last_assigned_at`
- `last_seen_at`
- `accepting_sessions`
- `presence_status`
- `routing_weight`

### 11.2 会话分配信息

建议在访客或会话维度补充：

- `owner_id`：当前接待客服
- `route_status`：`pending` / `assigned` / `closed`
- `queue_name`
- `preferred_skill`
- `last_assigned_at`
- `last_transfer_at`
- `priority`
- `source_entry`
- `served_by_type`

MVP 阶段可以先复用 `visitor.to_id` 作为 owner，但语义要统一为“当前接待客服”，而不是“入口指定客服”。

### 11.3 事件主题

建议引入以下事件类型：

- `session.assign.requested`
- `session.assigned`
- `session.assign.timeout`
- `session.released`
- `message.created`

Kafka 可先继续复用单 topic 加 `type` 字段区分，后续再拆 topic。

## 12. 功能设计要求

除了分配本身，MVP 还要考虑主流客服系统中常见的功能组织方式。

### 12.1 会话生命周期

至少支持以下状态：

- `pending`
- `assigned`
- `transferring`
- `waiting_reply`
- `closed`

### 12.2 客服工作台

工作台不再只是“在线访客列表”，而要逐步演进为：

- 我的会话
- 待分配会话
- 排队会话
- 最近会话
- 客服负载概览

### 12.3 路由配置

至少支持：

- 默认分配模式
- 技能标签
- 容量上限
- 粘性分配开关
- 扩散超时阈值
- 默认公共队列

### 12.4 审计与观测

至少记录：

- 谁在什么时间被分配给谁
- 谁做了转接
- 会话等待了多久
- 哪类会话最常扩散失败
- 各客服负载分布

## 13. 接口与页面改动

### 13.1 访客侧

- `/livechat` 支持无 `user_id` 打开
- `/visitor_login` 支持：
- `entry_id`
- `service_line`
- `visitor_id`
- `extra`

兼容要求：

- 保留 `to_id` 和 `user_id` 兼容逻辑
- 无旧参数时，自动走智能分配

### 13.2 工作台

工作台需展示：

- 当前在线会话
- 当前排队会话
- 每个客服当前负载
- 每个客服剩余容量
- 技能标签

### 13.3 管理配置

MVP 阶段至少支持配置：

- 客服最大接待数
- 技能标签
- 默认分配模式
- 扩散超时阈值

## 14. 迁移与重构策略

这次重构允许调整现有结构，但要避免一次性推翻导致系统不可用。

### 14.1 重构方式

- 先抽象路由中心，再迁移入口
- 先兼容旧参数，再切主路径
- 先补会话状态和读模型，再逐步替换工作台数据来源
- 先保留旧字段，再逐步收敛语义

### 14.2 建议迁移顺序

1. 明确路由接口和会话状态机
2. 接入无 `user_id` 的统一入口
3. 让新入口走路由中心
4. 让旧入口走兼容适配层
5. 工作台改为读取会话读模型
6. 再继续收缩老逻辑

## 15. 实现步骤

### Phase 1

- 新增 `MVP` 文档
- 明确入口兼容策略
- 抽象人工客服路由器接口
- 定义会话状态机、队列模型和 owner 语义

### Phase 2

- 访客入口支持无 `user_id`
- 新增人工客服分配逻辑
- 实现在线、容量、技能、粘性、最小负载分配
- 抽离客服状态管理与路由中心
- 让控制器和 WebSocket 改为调用路由中心，而不是直接分配

### Phase 3

- 引入排队与超时扩散
- 打通 Kafka 分配事件
- 工作台补充分配状态展示
- 工作台明确区分“接管 pending 会话”和“转接已接待会话”，避免用离线/关闭语义误处理活跃会话
- 引入基础读模型和会话摘要更新

### Phase 4

- 梳理消息事件模型
- 增加摘要视图或状态缓存
- 为统计、AI、质检预留读扩散接口
- 收敛旧的 `user_id/to_id` 直连思路

## 16. 验收标准

满足以下条件视为 MVP 可交付：

- 访客打开 `/livechat` 时，不传 `user_id` 也能完成会话建立
- 系统可以自动选择一个在线且未满载的客服
- 同一访客再次进入时，优先回到原客服
- 技能池无人可接待时，系统能进入排队并在阈值后扩散
- 客服结束会话后，系统会释放其占用容量
- 工作台可以看到当前会话 owner 和客服容量状态
- 工作台能区分待分配、进行中和最近会话
- 消息主记录不因扩散而重复写多份
- Kafka 消费失败不会阻塞访客主链路，并具备超时与重试退避
- 主要路由逻辑不再散落在控制器和 WebSocket 分支中

## 17. 非功能要求

- 所有外部依赖调用必须有超时控制
- 分配链路失败不能影响访客消息主链路可用性
- 分配事件至少做到可观察、可重试、可回退
- 旧 URL 和旧接口要兼容一段时间，避免前端嵌入脚本立刻失效
- 设计上优先阶段性重构，但允许对核心路由模型做必要的整体调整
- 新架构要优先考虑后续扩展，而不是只满足当前一个分配场景

## 18. 风险与注意事项

- 当前系统很多地方默认 `visitor.to_id` 就是客服账号，改造时需统一语义
- 现有在线客服状态主要依赖 WebSocket 在线感知，后续需确认是否足够稳定
- 分配器如果同时承担人工客服和 agent/机器人两种角色，语义会混乱
- 若直接把写扩散做进主消息链路，容易放大失败面，应优先异步化
- 旧版嵌入脚本和外部站点可能仍依赖 `user_id` 参数，兼容期必须保留
- 如果只是补几个分支判断而不抽象路由中心，后续复杂度会继续失控
- 如果过早删除旧字段和旧入口，现网嵌入页面会有较高兼容风险

## 19. 本次实现建议

按风险最低的顺序，优先做：

1. 文档落地
2. 路由中心与会话状态建模
3. 统一入口兼容设计
4. 人工客服智能分配
5. 排队与超时扩散
6. 消息视图扩散和工作台重组

本仓库后续开发以本文件为 MVP 基线，如实现中出现与当前代码结构冲突的地方，以“兼容现有链路、小步可验证演进”为优先原则处理。
